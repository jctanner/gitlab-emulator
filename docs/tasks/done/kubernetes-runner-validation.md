# Kubernetes Runner Validation Plan

Status: VM-service runner and in-cluster runner both implemented and validated
against the Vagrant VM stack.

This plan validates whether the official GitLab Runner Kubernetes executor can
run CI jobs against the emulator without requiring the emulator to execute jobs
itself.

For an integration-oriented description of all supported runner deployment
methods and registration flows, see `docs/runbooks/runner-deployment.md`.

## Goal

Prove this flow end to end:

1. A pipeline/job is created in the emulator.
2. An official `gitlab-runner` registered with the Kubernetes executor polls the
   emulator runner coordinator API.
3. The runner creates a job pod in a local k3s cluster.
4. The job pod executes the CI script.
5. The runner streams trace output, final status, and artifacts back to the
   emulator.
6. The admin UI and GitLab-shaped API surfaces show the runner, job, trace,
   status, and artifacts.

## Proposed VM Topology

Keep the existing emulator VM as the GitLab server and add one new validation
VM:

- `server`
  - Existing emulator stack.
  - Serves `https://glemu.local`.
  - Owns API, Git HTTP, trace, artifact, cache, and admin UI endpoints.
- `runner`
  - Existing Docker executor validation runner.
  - Remains the baseline official runner validation path.
- `k8s-runner`
  - New VM.
  - Runs single-node k3s.
  - Runs official `gitlab-runner` as a system service for the first Kubernetes
    executor validation.
  - Also hosts a second official `gitlab-runner` manager as a Deployment inside
    k3s for in-cluster validation.
  - Both runners use the Kubernetes executor to create CI job pods inside local
    k3s.

The VM-service runner proves coordinator compatibility with the Kubernetes
executor. The in-cluster runner proves the production-shaped manager-pod mode
where the runner process itself runs in Kubernetes.

## Network Flow

```text
emulator/server VM
  https://glemu.local
      ^
      | runner coordinator API, Git HTTP, traces, artifacts
      |
k8s-runner VM
  gitlab-runner service
      |
      | Kubernetes API
      v
  k3s job pods

  gitlab-runner manager pod
      |
      | Kubernetes API via service account
      v
  k3s job pods
```

Required network setup:

- `k8s-runner` must resolve `glemu.local` to the emulator server VM IP.
- `k8s-runner` must trust the emulator Caddy root CA.
- Job pods must also be able to resolve and trust `glemu.local`; this is the
  likely extra validation point beyond the Docker executor.
- The in-cluster runner manager pod must also resolve and trust `glemu.local`.
- The k3s cluster must be able to pull the requested CI images.

## Initial Implementation Slices

### 1. Add the VM

Add a `k8s-runner` machine to `Vagrantfile`.

Suggested shape:

- Debian Bookworm base box.
- 2 CPU.
- 4 GB RAM minimum.
- Same libvirt network as `server` and `runner`.
- Hostname: `k8s-runner`.

### 2. Provision k3s

Provision the VM with:

- k3s single-node install.
- `kubectl` configured for the vagrant user or root.
- A `gitlab-runner` namespace.
- A service account, role, and role binding compatible with GitLab Runner
  Kubernetes executor pod creation.

Keep this minimal. The first goal is one local cluster that can create pods,
not a production-grade cluster install.

### 3. Install Trust and Name Resolution

Mirror the existing runner trust setup:

- Copy the current emulator Caddy root CA from the server container.
- Install it into the `k8s-runner` VM trust store.
- Add `glemu.local` to `/etc/hosts` on the VM.
- Ensure job pods also trust the CA and resolve `glemu.local`.

Pod trust may require one of these approaches:

- mount a ConfigMap/Secret containing the CA into job pods and set the runner
  Kubernetes executor CA options;
- bake a small validation image with the CA installed;
- configure k3s/CoreDNS and runner pod settings so the job pod can reach
  `https://glemu.local` cleanly.

Prefer the least invasive option that works with the official runner.

### 4. Register the Runner

Register official `gitlab-runner` against the emulator:

- URL: `https://glemu.local`
- Registration token: emulator runner registration token.
- Executor: `kubernetes`
- Namespace: `gitlab-runner`
- Tags: `k8s`
- Run untagged: probably false for the first validation so jobs are routed
  deliberately.

Expected runner config:

```toml
[[runners]]
  name = "glemu-k8s-runner"
  url = "https://glemu.local"
  executor = "kubernetes"
  [runners.kubernetes]
    namespace = "gitlab-runner"
```

The exact config should be generated by `gitlab-runner register` where
possible, then adjusted only as needed for k3s and emulator CA behavior.

### 5. Add Make Targets

Add operational targets alongside the existing VM runner targets:

- `make vm-k8s-runner-up`
- `make vm-k8s-runner-status`
- `make vm-k8s-runner-install-ca`
- `make vm-k8s-runner-register`
- `make vm-k8s-runner-validate`
- `make vm-k8s-runner-logs`
- `make vm-k8s-runner-pods`

The validation target should create or reuse a tagged pipeline job and verify:

- runner registration is visible in `/api/v4/runners`;
- `/admin/runners` shows the k8s runner;
- a `tags: [k8s]` job is assigned to that runner;
- k3s creates a job pod;
- trace output reaches the emulator;
- final job status reaches the emulator;
- artifacts upload and can be downloaded.

### 6. Add a Tagged Validation Pipeline

Use a small `.gitlab-ci.yml` designed for fast Kubernetes executor validation:

```yaml
k8s_probe:
  image: alpine:3.20
  tags:
    - k8s
  script:
    - echo k8s runner started
    - uname -a
    - mkdir -p out
    - echo k8s artifact > out/result.txt
  artifacts:
    paths:
      - out/result.txt
```

This should stay separate from the Docker runner validation job so failures are
easy to attribute.

## Emulator API Surfaces Under Test

The Kubernetes executor should use the same runner coordinator surfaces already
used by the Docker executor:

- `POST /api/v4/runners`
- `POST /api/v4/runners/verify`
- `POST /api/v4/jobs/request`
- Git HTTP clone/fetch with CI job token
- `PATCH /api/v4/jobs/:id/trace`
- `PUT /api/v4/jobs/:id`
- `POST /api/v4/jobs/:id/artifacts`
- cache upload/download endpoints, if the validation job uses cache

Potential Kubernetes-specific compatibility gaps may appear in:

- runner metadata fields sent during registration/verify/poll;
- job payload fields expected for Kubernetes executor pod setup;
- TLS/clone behavior from inside the job pod;
- artifact upload content type or multipart behavior;
- trace patch offset/range behavior under longer logs.

## Acceptance Criteria

The first validation slice is complete when:

- `vagrant up k8s-runner` provisions a working k3s node.
- `make vm-k8s-runner-register` registers an official runner using
  `executor = "kubernetes"`.
- `/admin/runners` lists the runner with executor `kubernetes`.
- A `tags: [k8s]` pipeline job moves from `pending` to `running` to `success`.
- `kubectl get pods -n gitlab-runner` shows the CI job pod being created.
- Job logs stream into the emulator and are visible from the UI/API.
- Artifacts upload to the emulator and can be downloaded.
- The Docker runner validation path still works.

Validated on 2026-06-26:

- `vagrant up k8s-runner` provisioned Debian Bookworm with k3s
  `v1.35.5+k3s1` and GitLab Runner `19.1.1`.
- `make vm-k8s-runner-register` registered an official runner using
  `executor = "kubernetes"`.
- `make vm-k8s-runner-validate` passed.
- The validation job `k8s_probe` moved from `pending` to `running` to
  `success`.
- k3s created a runner pod in the `gitlab-runner` namespace.
- Trace markers reached the emulator.
- Artifact metadata was recorded and exposed by the job API.
- The emulator now creates a distinct persisted runner token for each

## In-Cluster Runner Slice

Now that the VM-service runner works, validate running the runner itself inside
k3s.

Implementation target:

- deploy `gitlab-runner` as a Kubernetes Deployment in
  `gitlab-runner-incluster`;
- use a dedicated service account, role, and role binding for job pod
  management;
- register with the emulator registration token and tag `k8s-incluster`;
- mount the emulator Caddy root CA into the runner manager pod;
- configure runner manager pod and job pods with `glemu.local` host aliases;
- validate with a tagged `k8s-incluster` pipeline job.

Operator targets:

- `make vm-k8s-incluster-deploy`
- `make vm-k8s-incluster-status`
- `make vm-k8s-incluster-logs`
- `make vm-k8s-incluster-pods`
- `make vm-k8s-incluster-validate`

Done when:

- the runner manager Deployment rolls out successfully;
- `/admin/runners` lists the in-cluster runner;
- a `tags: [k8s-incluster]` job moves from `pending` to `running` to
  `success`;
- `kubectl get pods -n gitlab-runner-incluster` shows both the manager pod and
  CI job pods;
- trace markers and artifact metadata round-trip through the emulator.

Validated on 2026-06-26:

- `make vm-k8s-incluster-validate` passed.
- The manager Deployment `glemu-k8s-incluster-runner` rolled out in namespace
  `gitlab-runner-incluster`.
- The validation job `k8s_probe` used tag `k8s-incluster` and moved from
  `pending` to `running` to `success`.
- k3s showed both the manager pod and a separate CI job pod in
  `gitlab-runner-incluster`.
- Trace markers and artifact metadata were recorded by the emulator.

Validated on 2026-06-27:

- `make vm-k8s-incluster-secret-file-test` passed.
- File-mode secrets were readable from a job pod created by the in-cluster
  runner manager and were masked in stored traces.
